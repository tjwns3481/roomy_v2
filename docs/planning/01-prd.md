# PRD (제품 요구사항 정의서)

> Roomy: 한국 숙박 시설을 위한 디지털 게스트 가이드북 SaaS

---

## MVP 캡슐

| # | 항목 | 내용 |
|---|------|------|
| 1 | 목표 | 한국 펜션/글램핑 호스트를 위한 디지털 가이드북 서비스 MVP 출시 |
| 2 | 페르소나 | 에어비앤비/펜션 운영자 (반복 문의 줄이고 싶은 호스트) |
| 3 | 핵심 기능 | AI 가이드북 자동 생성 (에어비앤비 링크 → 한국어 가이드북) |
| 4 | 성공 지표 (노스스타) | 월 매출 500만원 (유료 구독자 ~100명) |

## 1. 문제 정의

### 1.1 현재 상황 (As-Is)

숙박 호스트들의 고충:
- 게스트마다 **같은 질문 반복** (WiFi 비밀번호, 도어락, 보일러 사용법)
- **전화/카톡 문의 폭주** → 업무 방해, 스트레스
- 종이 안내서는 **유지보수 어려움** + 환경 문제
- 기존 솔루션(Touch Stay)은 **한국어 미지원**, 해외 결제만

### 1.2 목표 상황 (To-Be)

- 에어비앤비 링크만 입력하면 **AI가 한국어 가이드북 자동 생성**
- 게스트는 **모바일 웹**으로 편리하게 정보 확인
- 호스트는 **반복 문의 90% 감소** → 본업에 집중
- **완전한 한국어 UI** + 한국 특화 콘텐츠 (분리수거, 온돌, 도어락)

### 1.3 왜 지금인가? (Why Now)

- Touch Stay가 **한국 시장 진출 전** (한국 PMS 연동 없음)
- AI 기술 성숙 → **자동 가이드북 생성** 가능
- 펜션/글램핑 시장 **성장세** (코로나 이후 국내 여행 증가)

## 1.5 벤치마킹 분석

### 유사 서비스 분석

| # | 서비스명 | 핵심 기능 | 성공/실패 요인 | 우리와의 차이점 |
|---|----------|----------|---------------|---------------|
| 1 | Touch Stay | 디지털 게스트 가이드북 SaaS | 성공: 글로벌 선두주자 / 단점: 한국어 미지원 | 영어 UI vs 완전 한국어 |
| 2 | Hostfully | 가이드북 + PMS | 성공: 통합 솔루션 / 단점: 복잡함 | 복잡함 vs 단순함 |
| 3 | YourWelcome | 태블릿 기반 | 성공: 프리미엄 / 단점: 하드웨어 필요 | 하드웨어 vs 웹 기반 |

### 차별화 포인트

| 항목 | Touch Stay | Roomy | 왜 더 나은가? |
|------|------------|-------|-------------|
| 언어 | 영어 UI | 완전한 한국어 | 한국 호스트/게스트 UX 향상 |
| 가격 | $99/년 (~13만원) | ₩49,000/년 | 가격 경쟁력 3배 |
| 지도 | Google Maps | 네이버/카카오 지도 | 한국 사용자 친숙함 |
| 알림 | SMS/Email | 카카오 알림톡 (Phase 2) | 한국인 메신저 점유율 95% |
| 콘텐츠 | 해외 기준 | 한국 특화 (분리수거, 온돌) | 한국 숙소에 딱 맞는 템플릿 |

### 킬러 질문 대응
> "Touch Stay가 한국어 지원하면?"

우리는 **한국 특화 콘텐츠**와 **카카오톡 연동**으로 차별화. 글로벌 서비스가 분리수거 가이드, 보일러 사용법, 도어락 비밀번호 자동 노출 같은 한국 특화 기능을 제공하기 어려움.

## 2. 사용자 페르소나

### 2.1 주요 페르소나: 호스트

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "바쁜 펜션 사장님 김사장" |
| 나이/직업 | 45세, 제주 펜션 운영 (본업: 카페) |
| 핵심 니즈 | 게스트 문의 줄이고, 본업에 집중하고 싶음 |
| 불편 포인트 | 매일 같은 질문 (WiFi, 보일러), 전화 받느라 시간 낭비 |
| 사용 상황 | 예약 확정 → 가이드북 링크 카톡 발송 → 게스트가 셀프 확인 |

### 2.2 보조 페르소나: 게스트

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "스마트한 여행자 이여행" |
| 나이/직업 | 32세, 직장인 |
| 핵심 니즈 | 빠르게 정보 찾기, 호스트에게 전화하기 민망 |
| 불편 포인트 | 종이 안내서 읽기 귀찮음, 필요한 정보 찾기 어려움 |
| 사용 상황 | 체크인 전 링크 받음 → 모바일에서 WiFi/도어락 확인 |

### 2.3 페르소나의 하루 (시나리오)

**호스트 시나리오:**
김사장은 제주에서 펜션 3채를 운영하면서 카페도 운영한다. 매일 게스트에게서 "WiFi 비밀번호요", "보일러 어떻게 켜요?", "분리수거 어디서 해요?" 같은 문의가 10통 이상 온다. Roomy를 사용한 후, 예약 확정 시 가이드북 링크만 보내면 게스트가 스스로 확인한다. 문의가 90% 줄었고, 카페 운영에 집중할 수 있게 되었다.

**게스트 시나리오:**
이여행은 제주 펜션을 예약했다. 체크인 전날 카톡으로 가이드북 링크가 왔다. 모바일로 열어보니 도어락 비밀번호, WiFi, 주변 맛집까지 깔끔하게 정리되어 있다. 도착 후에도 보일러 사용법을 영상으로 확인할 수 있어서 호스트에게 전화할 필요가 없었다.

## 3. 사용자 스토리

### FEAT-0: 인증
- [ ] 호스트로서, 이메일/비밀번호로 회원가입할 수 있어야 한다. 왜냐하면 계정을 만들어 가이드북을 관리하고 싶기 때문.
- [ ] 호스트로서, 구글 소셜 로그인을 할 수 있어야 한다. 왜냐하면 빠르게 시작하고 싶기 때문.

### FEAT-1: 가이드북 생성 (MVP 핵심)
- [ ] 호스트로서, 에어비앤비 링크를 입력하면 AI가 가이드북을 자동 생성해야 한다. 왜냐하면 직접 작성하기 귀찮기 때문.
- [ ] 호스트로서, 수동으로 가이드북을 생성할 수 있어야 한다. 왜냐하면 에어비앤비가 아닌 숙소도 있기 때문.
- [ ] 호스트로서, 블록 에디터로 가이드북을 편집할 수 있어야 한다. 왜냐하면 AI가 생성한 내용을 수정하고 싶기 때문.

### FEAT-2: 게스트 뷰
- [ ] 게스트로서, 모바일 웹에서 가이드북을 볼 수 있어야 한다. 왜냐하면 앱 설치 없이 바로 보고 싶기 때문.
- [ ] 게스트로서, 핵심 정보(WiFi, 도어락, 체크인/아웃)를 빠르게 찾을 수 있어야 한다. 왜냐하면 시간이 없기 때문.
- [ ] 게스트로서, 주변 맛집/관광지 정보를 볼 수 있어야 한다. 왜냐하면 여행 계획에 도움이 되기 때문.

### FEAT-3: 대시보드
- [ ] 호스트로서, 내 가이드북 목록을 볼 수 있어야 한다. 왜냐하면 여러 숙소를 관리하기 때문.
- [ ] 호스트로서, 가이드북 조회수를 볼 수 있어야 한다. 왜냐하면 게스트가 실제로 보는지 확인하고 싶기 때문.

### FEAT-4: 공유 (MVP)
- [ ] 호스트로서, URL을 복사해서 게스트에게 공유할 수 있어야 한다. 왜냐하면 카톡으로 보내고 싶기 때문.
- [ ] 호스트로서, 이메일로 가이드북 링크를 발송할 수 있어야 한다. 왜냐하면 예약 시스템에서 이메일을 사용하기 때문.

### FEAT-5: 카카오 알림톡 연동 (Phase 2)
- [ ] 호스트로서, 카카오 알림톡으로 자동 발송할 수 있어야 한다. 왜냐하면 게스트가 카톡을 가장 많이 쓰기 때문.

## 5. 범위 관리

### 5.1 MVP (반드시 포함)
- FEAT-0: 이메일/구글 로그인
- FEAT-1: AI 가이드북 생성 (에어비앤비 링크) + 블록 에디터
- FEAT-2: 모바일 게스트 뷰

- 다중 이미지 갤러리
- 지도 연동 (네이버/카카오맵)
- AI 챗봇 (게스트 질문 자동 응답)

- 카카오 알림톡 (사업자등록 필요, Phase 2)
- 야놀자/여기어때 PMS 연동 (Phase 3)
- 네이티브 앱 (모바일 웹으로 충분)
- 다국어 지원 (한국어만)

## 7. 해결 가설과 실험

### 7.2 실험 루프

| 실험 | MVP 출시 후 100명 호스트에게 무료 체험 제공 |
